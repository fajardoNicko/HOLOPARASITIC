Holoparasitic Vascular Percolation

Program Flowcharts

The whole codebase expressed as five standard flowcharts — every symbol drawn from the source, not from the prose. Read them in order: Fig 1 is the driver, Figs 2–4 are the routines it calls, Fig 5 is the sweep that calls Fig 2 twenty times over.

Fig 1Main program — scripts/run_full.py

The driver. Everything before the loop happens exactly once; the six production stages then run inside a guarded loop so that a failure in a late stage cannot destroy work already written to disk.

Reading the loop: each stage is wrapped in try/except. On an exception the driver prints [n] FAILED and moves on; on success _checkpoint() rewrites data/results.npz and data/summary.json including last_completed_stage — which is what makes an interrupted run resumable rather than restartable.

Fig 2Monte Carlo damage loop — percolation.density_response()

The heart of the program and where 106 trials happen. Note that the graph is converted to a CSR adjacency matrix before the outer loop — nothing inside either loop touches a NetworkX object. That one decision is what makes a million trials a coffee break instead of a cluster job.

Two nested loops: the outer runs over 50 infestation loads ρ; the inner runs 20 000 trials at each load. 50 × 20 000 = 1 000 000 rows of (ρ, collapse-bit), which is exactly what Fig 3 consumes.

Fig 3Threshold estimation — percolation.fit_pc() → bootstrap_pc()

Turns a million binary outcomes into one number with an error bar. The two guards on the right and left of the spine are the honesty of this routine: it refuses to invent a threshold when the data does not contain one.

Why the fallback exists: the logistic inflection −intercept/coef can extrapolate outside [0,1] when collapse is already near-certain at the edge of the sweep. When that happens the linearly interpolated empirical 50 % crossing is the honest estimate. A genuine NaN means the curve never reaches 50 % anywhere in ρ ∈ [0,1] — a host the sink simply cannot collapse — and that is reported, not filled in.

Fig 4Hydraulic solve and cavitation — hydraulics.HydraulicSolver.solve()

The physics core, and the only place the parasite exists as a number. The haustorium is one term: a leak conductance h[i] tying a host node to a reservoir at −3 MPa. The decision diamond at the bottom is the hinge of the whole study — it is where a hydraulic event becomes a topological one.

The grounding term: diag(h + 1e−12) keeps A non-singular even when severe infestation detaches part of the network. Without it spsolve would return a plausible, meaningless answer to a singular system — a silent failure, the worst kind.

Fig 5Parameter sweep — sweep.pc_surface()

Two nested loops over host architecture (D, varied through reticulation density at constant node count) and parasite aggressiveness (Kh). Every cell re-runs the entire pipeline — Figs 2, 3 and 4 in full. Nothing here is interpolated.

Why the NaN branch is drawn and not hidden: some cells have no threshold at all — a network so densely reticulated that a weak sink cannot collapse it, ever. Recording those as NaN rather than dropping them keeps the model's behaviour visible exactly where it is most interesting: the region where the host wins.

Two places where these flowcharts disagree with METHODOLOGY.md — reconcile before submission.
1 · Site, not bond, percolation. The prose says ρ is "the fraction of vessels drained or disabled" (edges). Fig 2 shows what the code does: _trial_random computes idx = np.flatnonzero(rng.random(n) > rho) with n = adj.shape[0] — that is node (junction) removal, classic site percolation. Same critical behaviour and same qualitative conclusions, but not the same number.
2 · Root attachment is not enforced. The prose defines the giant component as "the largest surviving subnetwork that remains connected to the root." _gcc_size returns np.bincount(labels).max() — the largest component anywhere, with no root check. On a network this reticulated the two nearly always coincide, but the claim is not something the code currently guarantees.
What is not yet drawn, because it is not yet written. There is no scripts/run_empirical.py. It would read empirical/measurements/*.csv, run the same fractal.box_counting_dimension() over venation_topdown/, join each species' measured D to the pc_vs_D curve already sitting in results.npz, and emit four tests via scipy.stats: one-sample t-test of D_real vs D_model, one-way ANOVA of D across the five species, log-log regression testing H₀ that Murray's exponent = 3, and the central Pearson correlation of D_measured against p_c_predicted. Every dependency it needs already exists.
